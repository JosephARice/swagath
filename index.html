12 Steps to AI-Driven Wikis on GitHub Pages

  1. Create GitHub Repository: Start by creating a new public repository on GitHub to host your wiki files.
  2. Choose Static Site Generator (SSG): Select an SSG compatible with GitHub Pages (like Jekyll - native support, Hugo, Eleventy, or even plain HTML/JS). This tool will build your HTML pages from source files (like .md files).
  3. Local Development Setup: Clone the repository to your local machine and set up the chosen SSG's development environment according to its documentation.
  4. Develop Basic Wiki Structure: Create your initial wiki pages using Markdown (.md files). Organize them logically with folders and create basic navigation (e.g., a sidebar, index page).
  5. Enable GitHub Pages: Go to your repository's Settings -> Pages. Select the branch (usually main or gh-pages) and folder (usually / or /docs) to deploy from. GitHub Pages will build and host your SSG site.
  6. Select AI Feature & Provider: Decide what AI feature you want (e.g., semantic search, Q&A over documents, summarization). Choose an AI provider (like OpenAI, Google Gemini, Cohere) and identify the relevant API.
  7. Obtain API Key: Sign up for the chosen AI provider's service and get an API key. Crucially, plan how to handle this key securely – exposing it directly in client-side code is risky. (Consider environment variables during build or a serverless function proxy, though the latter adds complexity beyond basic GH Pages).
  8. Design AI Interface: Add UI elements to your wiki's template(s) for the AI interaction (e.g., a search input box, a button to submit a question, an area to display results).
  9. Implement Client-Side JavaScript: Write JavaScript code (in your SSG's assets folder) to:
    • Capture user input from the UI elements.
    • Prepare the data/prompt for the AI API call.
    • Use the fetch API to make requests to the AI provider's endpoint, including your (securely handled) API key.
  10. Process & Display AI Results: Enhance your JavaScript to parse the JSON response from the AI API and dynamically update the wiki page's HTML to display the results (e.g., show search results, display an answer).
  11. Test and Refine: Test the AI integration thoroughly. Debug issues, refine the prompts sent to the AI for better results, and improve the user interface based on testing. Test deployment on GitHub Pages.
  12. Commit, Push, and Iterate: Commit your code (SSG configuration, content, templates, JavaScript) to your repository and push it to the branch configured for GitHub Pages. Your site will rebuild and deploy. Continuously add content and improve the AI features.

Linking Between Wiki Pages Using relative links is essential for creating a functional multi-page wiki on GitHub Pages. The structure of your repository will determine the paths you use in your links. Consider the following example file structure:   wiki-repo/ ├── index.html ├── section1/ │ └── page1.html └── section2/ └── page2.html To link from the homepage (index.html) to page1.html, which is inside the section1 directory, you would use the following link in your index.html: Go to Page 1. From page1.html, to link back to the homepage (index.html), you would use: Home, as ../ directs the browser to go up one level in the directory structure. To link from page1.html to page2.html, which is in the section2 directory (a sibling of section1), you would use: Go to Page 2. Here, ../ goes up from section1 to the root, and another ../ would be incorrect; instead, you would then navigate down to the section2 directory. GitHub Pages automatically handles the .html extension in your links. This means that even though your files are named with the .html extension, you typically don't need to include the extension in your relative links. For example, About Us would correctly link to about.html if it exists in the same directory. This simplifies the linking process and makes the URLs appear cleaner.   7. Deploying Your HTML Wiki to GitHub Pages Once you have created your HTML pages, embedded the YouTube videos, set up the navigation, and applied any desired CSS styling, the final step is to deploy your wiki to GitHub Pages. This involves several steps:   Commit and Push Your Files: First, ensure that all your HTML and CSS files (including the index.html file) are saved, committed to your local Git repository, and pushed to your remote GitHub repository. You can use Git commands like git add., git commit -m "Initial wiki content", and git push origin main (assuming main is your default branch).   Enable GitHub Pages: Next, navigate to your repository on the GitHub website. Click on the "Settings" tab. In the left sidebar, under the "Code and automation" section, click "Pages". Under the "Source" section, select "Deploy from a branch". Choose the branch you want to deploy (typically main or master). Then, select the folder containing your HTML files. If your index.html file is in the root of your repository, choose "(root)". If you have placed your HTML files in a docs folder (a common practice for documentation sites), select the /docs option. Finally, click the "Save" button.   Access Your Deployed Wiki: After saving the settings, GitHub will start the process of building and deploying your website. Once the process is complete (which might take a few minutes), your wiki will be accessible at a unique URL. If you created a project page (repository name is not in the format .github.io), the URL will be https://.github.io//. If you created a user or organization site (repository named .github.io or .github.io), the URL will be https://.github.io/ or https://.github.io/ respectively. You might need to wait a few minutes for the site to become live.   It's also worth noting that you have the option to configure a custom domain name for your GitHub Pages site if you prefer a more personalized web address. This involves adding a CNAME file to your repository and configuring your domain registrar's DNS settings.   10. Important Considerations and Limitations While GitHub Pages provides a convenient platform for hosting static wikis, it's important to be aware of certain considerations and limitations. GitHub Pages is primarily designed for hosting static content. This means that dynamic features that require server-side processing, such as a built-in search functionality that indexes the wiki content or user authentication, are not directly supported. However, you can potentially integrate client-side search solutions or embed external services to achieve similar functionality.   There are also usage limits to consider, such as the recommended repository size, the maximum size of the published site, and bandwidth limitations. Embedding a large number of high-resolution YouTube videos could contribute to the overall size of your wiki. Additionally, the embedded YouTube videos rely entirely on the availability of those videos on the YouTube platform. If a video is removed from YouTube, it will no longer be accessible within your wiki. Furthermore, content embedded via iframes might not be readily available for offline access unless the user has previously viewed the content and it has been cached by their browser. Finally, it's important to remember that GitHub Pages is not intended for commercial transactions or for hosting online businesses or SaaS applications.   11.
